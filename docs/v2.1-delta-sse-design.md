# v2.1 Delta-SSE — design + spike notes

> 状态: APPROVED (2026-05-13). 5 个决策点用户已拍板，PR D1 可以开。

## 1. 现状

当前 read pipeline:

```
CC 写 jsonl 一行
  → chokidar (1Hz 节流) fire change
  → server broadcast SSE `invalidate { sessionId, kind: 'main' }`
  → client 收 → refreshSession(sid) (with rc.2 dedup + coalesce)
  → GET /api/sessions/<sid>
  → server: locateJsonl + findForkClosure + LRU cache (key = closure mtimes)
    → mtime 变了就 reparse 整个 closure (loadMergedChatFlow)
  → 返回 lite ChatFlow (workflow.nodes/edges 剥掉 + summary 内联)
  → ~120MB session = ~16.8MB payload / ~4.2s
  → client store applyChatFlow diff-merge (sessionSlice.ts:105 内嵌结构相等判定)
  → React re-render
```

**已有的二级优化**:
- Lazy ChatFlow B5：workflow.nodes / edges 不在 lite 里，每个 ChatNode 单独按需 `/chatnodes/workflows?ids=`
- rc.2 dedup + coalesce：保证最多一个 in-flight refresh per session
- Vite proxy timeout 60s：防止 4.2s 响应被中途 502

**瓶颈位置**:
1. 每次 invalidate 都要 server 重新序列化全 lite payload 推 16.8MB
2. 即使有 dedup，1Hz cadence × 4.2s 响应仍意味着 streaming turn 期间客户端始终落后 1 个周期
3. 浏览器拿到响应后 store diff-merge 也要遍历全 ChatNode 数组
4. 200K-line jsonl 会让 lite payload 增长到不可接受

## 2. 目标

**核心**: SSE 推 **delta**（增量记录），client 在本地 ChatFlow 上 patch，不再每次拉全文。

**保留**: fs.watch 作为 drift detector — 定期 / 启动时 / 客户端怀疑漂移时强制一次 full refresh，作为 sanity check。

**伸展目标**:
- 200MB+ session 也能 sub-second 反馈
- streaming turn 期间客户端是实时跟上 jsonl 写入的（不是节流到 1Hz）

## 3. 设计核心抉择

### 3.1 Delta 颗粒度

| 选项 | 描述 | 优点 | 缺点 |
|---|---|---|---|
| A | 每条 jsonl record 一个 delta | 最简单，server 1-pass 解析 | 单条 record 信息不全（缺 parent/tool 关联），client 端 reducer 复杂 |
| B | "coherent batch" | 一组关联 record 同发（如一个完整 turn） | 较实，但 streaming 不够实时（要等 stop_reason） |
| **C** | **高层语义事件**（chatnode-added / worknode-added / worknode-updated / chatnode-summary-updated 等） | server 把脏数据细节包起来；client reducer 只接合规对象 | server 要维护"上次推送的快照"做 diff，多一份内存 |

**推荐 C**。已有 LRU 缓存里就是 parsed ChatFlow，本质上 server 已经"知道全文"了，只是没在 delta 颗粒度上 expose。

### 3.2 状态权威 + drift 检测

- Server 持每 active session 的 **last-pushed snapshot**（per SSE subscriber? per session? 见 3.3）
- 每次 chokidar fire → server reparse → diff against last-pushed → 推语义事件
- 每个 delta event 带递增 `seq` 字段
- Client 维持 `lastSeqBySession`；收 delta 时若 `seq !== lastSeq + 1` → drift detected → 调旧 `GET /api/sessions/<sid>` 全量回拉
- 周期性 `ping`（每 30s）只带 server-side 当前 seq，client 比较，不一致则 force refresh

### 3.3 Per-subscriber 还是 per-session 快照？

| 方案 | 描述 | trade-off |
|---|---|---|
| Per-subscriber | 每个浏览器 tab 自己一份"已推送" snapshot | 准确 + 各 tab 独立。但 N 个 tab × M 个 session 的内存爆炸 |
| **Per-session** | server 维护每 session 一份"权威已推 snapshot"，所有 subscriber 共用 | 内存 OK，但新 tab join 时拿不到"它自己缺哪段"——只能强制 full refresh 一次然后跟 |

**推荐 per-session + 新 tab 强制 initial full refresh**。新 tab 加入本来就要拿 initial state；走 GET 一次再切 delta 模式简单可靠。

### 3.4 Backwards compat / migration

- 第一阶段 `invalidate` SSE 事件 **不动**，client 仍可走老路
- 新 `delta` 事件加在 SSE channel 上，老 client 不知道这事件类型就忽略
- Client 端按一个 feature flag 切换 "delta-first" vs "invalidate-first"（默认 invalidate 先，灰度切 delta）
- 全量 ship 后老 `invalidate` 退化为 drift trigger

## 4. PR 拆分（提案）

**PR D1 — Server-side delta emitter（不破坏 client）**

- 新文件 `src/server/services/chatFlowDeltaEngine.ts`
- 每 active session 维护一个 `lastPushedSnapshot: { chatNodes: Map<id, ChatNodeSummary>, seq: number }`
- chokidar fire → reparse closure → 跟 snapshot diff → emit:
  - `delta: { type: "chatnode-added", chatNode, seq }`
  - `delta: { type: "chatnode-summary-updated", chatNodeId, summary, seq }`（streaming 中 tool_use count 涨）
  - `delta: { type: "chatnode-removed", chatNodeId, seq }`（rare — fork purge / trash）
  - `delta: { type: "checkpoint", seq }`（流末标记给 drift sanity）
- 更新 snapshot 到当前
- **仍然同时 broadcast `invalidate`** — client 不变
- Server unit tests：模拟一组 jsonl writes 验证 delta 序列正确

**PR D2 — Client-side delta apply**

- 新 store action `applyChatFlowDelta(sessionId, delta)`
- 实现 reducer 对应每种 delta type
- App.tsx 加 `delta` SSE handler → call applyChatFlowDelta
- 加 `lastSeqBySession` 跟踪、gap 检测、自动 fallback 到 full refresh
- 客户端 feature flag `useDeltaSSE` 默认 false
- 单测：用脚本生成的 delta 序列做 reducer 状态机测试

**PR D3 — Cutover + drift validator**

- `useDeltaSSE` 默认开
- 周期性 ping（每 60s）携带 server seq；client 不匹配则强制 full refresh
- 删 `invalidate` event 作为"主信号"，仅保留为 drift trigger
- 文档更新 + handoff

**PR D4（可选 stretch）** — server-side incremental parse

- 不再 reparse 整个 closure；维护 jsonl 文件 offset，只读新追加的字节
- diff 跳过对已解析记录的重新比对
- 这是最大的性能收益（reparse 200MB jsonl 本身就是几秒）；放最后做避免阻塞 D1-D3 ship

## 5. 风险

| 风险 | 缓解 |
|---|---|
| Server 内存涨（每 active session 一份 snapshot） | session 关闭时清；snapshot 只保 chatNodeSummary（不存 workflow.nodes） |
| Delta 顺序错乱（多线程 / 多事件并发） | server-side 用单线程事件循环 + seq 单调递增 |
| Client 应用 delta 时 race（用户操作 + delta 同时改 chatFlow） | 跟现有 `applyChatFlow` 一样走 zustand 单线 reducer |
| reparse 整个 closure 仍是性能瓶颈 | D4 解决；D1-D3 先 ship 验证语义层正确性 |
| Drift detection 失效（seq 一致但实际状态偏）| 周期性 full refresh 作 sanity check（每 5 min 自动来一次也不重？或仅在 idle 期来）|

## 6. 决策（已锁）

1. **Delta 颗粒度** = **(C) 语义事件**（`chatnode-added` / `chatnode-summary-updated` / `chatnode-removed` / `checkpoint`）。Server 维护"上次推送的 chatNodeSummary map"，每次 reparse 后 diff 出语义事件。
2. **Snapshot 范围** = **per-session**（server 一份，所有 SSE subscriber 共用；新 tab join 走一次 full refresh 再切 delta 模式）。
3. **PR 顺序** = **D1 → D2 → D3 → (D4 stretch)**。语义层先 ship，incremental parse 后续优化。
4. **D2 feature flag** = **default ON**（作者自己开发自己测，不需要 soft-launch 兜底）。
5. **Drift detection 周期** = **30s default + 可配置 + 0 表示禁用**。配置项加在 Settings → 会话运行 tab，跟 auto-defer 同一段。LoomscopePreferences 新字段 `driftDetectionSec: number` (default 30, range [0, 600])。

每个 PR 按 v2.0.1 PR A/B/C 风格：单文档 + 单测齐全 + commit message 解释清楚。
